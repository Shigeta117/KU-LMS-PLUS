対応して

---

機能

9. Supabase Realtime
   を使っていないので拡張機能でスクレイプしても自動反映されない
   Chrome拡張がUPSERTを完了しても、Webアプリ側はポーリングも Realtime
   Subscription もないため手動更新が必要。supabase.channel() で assignments
   テーブルを購読すると即時反映できる。

10. deleteAssignment 関数がUIに繋がっていない（デッドコード）
    supabase.ts:34-41 に削除関数があるが、どこからも呼ばれていない。使わないなら
    削除すべき。使うなら「削除」ボタンをカードに追加すべき（非表示とは別の概念）
    。

11. Service Worker がなく真のPWAとして機能していない
    next.config.ts にService Workerの設定がなく、オフラインキャッシュが一切効かな
    い。Chromeの「インストール可能」の条件（fetch
    ハンドラ付きSW登録）も未充足の可能性がある。next-pwa
    または手動SWで対応すべき。

12. ログイン画面でSign Upが一般公開されている
    login/page.tsx:12 で isSignUp を切り替えられるが、このアプリは関大生向けのプ
    ライベートツールのはず。誰でもアカウントを作れる状態になっている。招待制また
    はサインアップ非公開（Supabase側で無効化）を検討すべき。

13. 拡張機能のログイン状態がWebアプリと完全に独立している
    拡張機能は chrome.storage にトークンを保存し、Webアプリは Supabase SDK
    セッションを使う。同じアカウントを使っても「どちらでログイン中か」が分離して
    いてユーザーの混乱を招く。少なくとも設定ページで拡張機能のログイン状態を確認
    できると良い。

コード保守性

14. スクレイピングロジックが content.js と bookmarklet.js に二重管理
    extractCourseName, extractCourseId, parseDeadline, parseCoursePage,
    fetchFollowingJsRedirect
    の5関数がほぼ同一コードで両ファイルに存在する（bookmarkletはES5互換にしている
    という制約あり）。WebClass側の仕様変更時に両方の修正が必要になるリスクを //
    NOTE: 変更時は bookmarklet.js も同期すること コメントで最低限明示すべき。

15. layout.tsx の appleWebApp.title が "KMS+" のままでリネーム漏れ
    layout.tsx:11 で title: 'KMS+' となっているが、プロジェクトは KU-LMS+
    にリネーム済み。ホーム画面追加時に "KMS+" が表示される。

16. Supabase の認証情報が2ヶ所で管理されている
    SUPABASE_URL / SUPABASE_ANON_KEY が .env → supabase.ts と bookmarklet.js 直書
    きで二重管理。ブックマークレットの性質上やむを得ないが、Next.jsのビルド時に
    bookmarklet.js へ環境変数を埋め込む仕組み（例：Next.jsのAPI
    routeでbookmarkletを動的生成）にするとズレが起きにくい。

17. InstallPrompt.tsx で参照している slide-up アニメーションが globals.css
    に未定義
    InstallPrompt.tsx:62 で animate-[slide-up_0.3s_ease-out]
    を使用しているが、globals.css に @keyframes slide-up
    が書かれていない。Tailwind v4 の arbitrary animation
    構文はキーフレーム定義を別途必要とするため、バナーがアニメーションなしで表示
    されている（または無視されている）。

18. fetchAssignments() で select('\*') を使用している
    supabase.ts:14 で全フィールドを取得しているが user_id
    などUI非表示フィールドも含む。必要なフィールドを明示的に指定すると意図が明確
    になり、スキーマ変更時の安全性も上がる。

19. AuthGuard の認証チェックと購読が並行管理
    AuthGuard.tsx:12-26 で getSession() の一度きりチェックと onAuthStateChange
    のリアルタイム購読を両方持っている。getSession() の結果と Realtime
    イベントが競合した場合の状態管理が複雑になっている。Supabaseの
    onAuthStateChange は初回イベントも発火するので、getSession()
    を省いてSubscriptionのみに統一できる。

20. page.tsx 内にロジックとレンダリングが混在したモノリシック構成
    AssignmentList コンポーネントが270行超で、データ取得・フィルタリング・楽観的
    更新・ログアウト・プルリフレッシュを全部一箇所に持つ。データ取得ロジックを
    useAssignments カスタムフックに分離するとテスタビリティと可読性が上がる。

    layout.tsx:31 で href="/icons/icon-192.png" を指定しているが、iOS推奨は

セキュリティ

23. ブックマークレットのAnon Keyがパブリックにアクセス可能
    public/bookmarklet.js はVercelで誰でもダウンロードできる。Anon Key はクライア
    ントサイド公開前提（RLSで保護）とはいえ、RLSポリシーの設定漏れが直接
    impersonation に繋がる。RLSの設定内容を docs/
    等に文書化して意図を明示することを推奨。

24. background.js のコメント「配布前にここを書き換えてください」が残っている
    background.js:4-9 の配布用コメントが実際の値に書き換えた後も残っている。コー
    ドの意図と実態が乖離しておりメンテナンス上の混乱を招く。
